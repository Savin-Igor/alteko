# Модуль: Mājas gatavības novērtējums / Оценка готовности дома

## Назначение

Центральный модуль платформы. Все остальные модули питают его данными и читают его результаты.

**Цель:** превратить разрозненные данные о доме (открытые источники + загруженные жителями + загруженные правлением) в один сводный показатель — **Building Readiness Score** — с конкретным `next_best_action` для пользователя.

Это **не** energy score. Energy score — один из восьми компонентов. Building Readiness Score измеряет **готовность дома как проекта**, не только энергоэффективность.

---

## Восемь компонентов Score

Каждый компонент — независимая оценка от 0 до 100, либо категориальный статус. Главное поле — `nextBestAction: string`.

### 1. Energy score (Energoefektivitātes vērtējums / Оценка энергоэффективности)

**Что:** энергокласс из BVKB + расчёт kWh/m²/год отопления.

**Источник:** локальная синхронизация BVKB ежедневно.

**Шкала:**
- A+ / A → 100
- B → 85
- C → 70
- D → 55
- E → 40
- F → 25
- G → 10
- Нет данных → null + статус «требует энергосертификата»

### 2. Funding eligibility score (Finansējuma piemērotības vērtējums)

**Что:** оценка пригодности дома к каждому из 5 финансовых сценариев.

**Источник:** rules engine — версионируемая логика правил.

**Структура:** массив `{scenario, status, confidence, reasoning}`:
- `scenario`: SCF / ALTUM_REMONTA_AIZDEVUMS / BANK / OWN_FUND / MIXED
- `status`: ELIGIBLE / LIKELY_ELIGIBLE / UNLIKELY / NOT_ELIGIBLE / UNKNOWN
- `confidence`: low / medium / high
- `reasoning`: текст объяснения для UI

**Особенность:** SCF-правила сейчас **черновые**, основанные на тексте плана likumi.lv/ta/id/361681. Когда выйдут MK noteikumi (Q4 2026) — обновляем версию rules engine, статусы по всем домам пересчитываются автоматически.

### 3. Document readiness score (Dokumentu gatavības vērtējums)

**Что:** какие документы есть, каких не хватает.

**Чек-лист:**
- ✅ / ❌ Энергосертификат (актуальный, не старше 10 лет)
- ✅ / ❌ Технический паспорт здания
- ✅ / ❌ Технический отчёт о состоянии (tehniskā apsekošana)
- ✅ / ❌ Список собственников (актуальный, ≤6 мес.)
- ✅ / ❌ Учредительные документы biedrība (если есть)
- ✅ / ❌ Доверенность уполномоченного лица
- ✅ / ❌ Ранее принятые решения собственников
- ✅ / ❌ Согласия GDPR от собственников

**Score:** % выполненных пунктов.

### 4. Owner decision readiness score (Īpašnieku lēmumu gatavība)

**Что:** какие решения собственников уже приняты, каких не хватает для движения дальше.

**Семь типов решений** (см. `readiness-glossary.md` → DecisionType):
- `PREPARATION_DECISION` — решение о начале подготовки
- `REPRESENTATIVE_AUTHORIZATION` — назначение уполномоченного лица
- `DATA_COLLECTION_CONSENT` — согласие на обработку данных
- `ENERGY_AUDIT_DECISION` — решение об энергоаудите
- `PROGRAM_APPLICATION_DECISION` — решение об участии в программе
- `LOAN_DECISION` — решение о кредите
- `SUPPLIER_SELECTION_DECISION` — решение о выборе поставщика

**Логика:** для каждого решения — статус `MISSING / IN_CAMPAIGN / PASSED / EXPIRED`. Score = % принятых из обязательных для текущей фазы дома.

### 5. Financial feasibility score (Finansiālā dzīvotspēja)

**Что:** хватит ли деньги дома (после субсидии и кредита) на проект.

**Расчёт:**
- Стоимость проекта (по €/м² × площадь, нормы ALTUM)
- Грант (если применим) — оценочно
- Кредит (по ставке ALTUM remonta aizdevums или банка)
- Распределение на квартиру в €/мес.
- Сравнение с текущими счетами

**Score:** 100 = «комфортно укладывается в фонд ремонта», 50 = «потребуется повышение взноса», 0 = «без серьёзного гранта не проходит».

### 6. Supplier selection status (Piegādātāju atlases statuss)

**Что:** на каком этапе выбор поставщика.

**Статусы:**
- `NOT_STARTED` — не начат
- `RFP_SENT` — запросы отправлены
- `RFP_RESPONDED` — есть предложения
- `EVALUATING` — сравнение
- `SELECTED` — выбран (с протоколом)
- `CONTRACTED` — контракт подписан

**Score:** не используется напрямую, передаётся как enum.

### 7. Legal confidence status (Juridiskās ticamības statuss)

**Что:** уверенность в юридической чистоте текущего состояния.

**Статусы:**
- `DRAFT` — черновик, нет юр. проверки
- `NEEDS_REVIEW` — нужна проверка юристом
- `VALIDATED` — проверено

**Применяется к:** решениям собственников, доверенностям, протоколам, договорам.

### 8. Data confidence status (Datu ticamības statuss)

**Что:** откуда взяты данные о доме.

**Статусы:**
- `PUBLIC_DATA` — только публичные (VZD, BVKB)
- `USER_UPLOADED` — загружено пользователем (счета, CSV)
- `BOARD_VERIFIED` — подтверждено правлением
- `PROFESSIONAL_VERIFIED` — подтверждено специалистом

**Применяется на UI:** каждое число имеет «значок» источника. Это нивелирует риск, что пользователь поверит черновому расчёту как окончательному.

## Главное поле: `nextBestAction`

Тип: `string` (локализованный, LV основной).

**Логика генерации:**

```
если data confidence == PUBLIC_DATA только:
  → "Заказать Readiness Report для полного анализа"

если energy score < 25:
  если нет энергосертификата:
    → "Заказать актуальный энергосертификат через BVKB"
  иначе:
    → "Энергопотери очень высокие — рассмотреть пакет SCF"

если document_readiness < 50:
  → "Подготовить недостающие документы (см. чек-лист)"

если owner_decisions PREPARATION_DECISION == MISSING:
  → "Запустить кампанию решения собственников о начале подготовки"

если все решения принять кроме PROGRAM_APPLICATION_DECISION:
  если SCF window == OPEN:
    → "Подать заявку через mans.altum.lv"
  если SCF window == EXPECTED:
    → "Дом готов. Ждём открытия окна SCF (Q2 2027)"
  если ни одно окно не подходит:
    → "Рассмотреть ALTUM remonta aizdevums как альтернативу"

если supplier_selection == NOT_STARTED и program approved:
  → "Открыть Tender Room для выбора поставщиков"
```

В UI это **одна строка** с CTA-кнопкой. Пользователь не должен сам собирать восемь компонентов в выводы.

## Дополнительные показатели (Integrity Score)

Ниже основного Score — отдельный блок «**Caurredzamības rādītāji** / **Показатели прозрачности**»:

- `procurement_transparency_score` — насколько процесс выбора поставщика прозрачен (количество приглашённых, наличие независимых предложений, документы)
- `supplier_conflict_risk` — риск конфликта интересов (через граф связей Lursoft + IUB open data)
- `decision_quality` — насколько решения собственников оформлены корректно (кворум, формулировки, юр. чистота)
- `owner_understanding_score` — насколько собственники получили объяснения простым языком (через статусы кампаний)
- `document_completeness` — насколько комплект документов готов к подаче
- `price_benchmark_deviation` — насколько предложения подрядчиков отклоняются от бенчмарков

Эти показатели — **не для оценки соседей**, а для **самозащиты дома**. Это якорь конкурентного рва ALTEKO (см. concept.md).

## Mājas pase / Паспорт дома

Это представление дома в UI как «карточка с временной осью». Не отдельная сущность, а **интерфейсный взгляд** на собранные данные.

Содержит:
- Базовые данные (адрес, серия, год, площадь, энергокласс)
- Историю Readiness Score за все периоды наблюдения
- Все принятые решения собственников
- Все загруженные документы
- Все счета (нормализованные)
- Все взаимодействия с поставщиками
- Все события: создание, обновление, выгрузки в BIS, поданные заявки

**Стратегическое значение:** через 2–3 года это полноценный архив дома. Перенести его в другую платформу — затратно. Это создаёт стоимость переключения и формирует ров.

## API (планируется)

| Route | Назначение |
|-------|-----------|
| `GET /api/readiness/:cadastralCode` | Полный Readiness Score дома |
| `POST /api/readiness/:cadastralCode/recalculate` | Пересчёт Score (после изменения данных) |
| `GET /api/readiness/:cadastralCode/next-action` | Только `nextBestAction` |
| `GET /api/readiness/:cadastralCode/integrity` | Integrity Score блок |
| `GET /api/readiness/:cadastralCode/passport` | Mājas pase — полная история |

См. подробности в [architecture.md](../technical/architecture.md) и [data-model.md](../technical/data-model.md).

## Юридический комментарий

ALTEKO **не выдаёт юридических заключений**. Readiness Score — **расчётная оценка**, не сертификация дома. Все статусы Legal confidence ≠ VALIDATED маркируются на UI как требующие юр. проверки.

**Ответственность за принятие решения** — на собственниках и уполномоченном лице. ALTEKO предоставляет данные, инструменты и расчёты, не принимает решения за дом.
