# Дизайн ALTEKO

UX-документ: принципы, экраны, сценарии и состояния.

---

## Часть 1. Принципы дизайна

### Кто наши пользователи

ALTEKO — мультирежимный продукт. У нас четыре пользовательских профиля:

| Режим | Пользователь | Возраст | Технический уровень |
|-------|--------------|---------|---------------------|
| **Publiskais** | Любой посетитель, часто — сосед, узнавший о продукте по ссылке | 30–70 | Низкий |
| **Iedzīvotāja** | Владелец квартиры в советском доме | 35–75 | Низкий |
| **Valdes** | Председатель biedrība, mājas vecākais | 40–70 | Средний (привык к Excel и PDF) |
| **Speciālista** | apsaimniekotājs, ESCO, projektu vadītājs | 28–60 | Высокий (профессиональный SaaS) |

Дизайн строится для **слабого звена** — пожилого жителя в Iedzīvotāja режиме. Speciālista режим может быть плотнее и информативнее, но базовые принципы (читаемость, мобильный приоритет) не нарушаются нигде.

---

### Принцип 1. Одно действие — один экран (для жильцов)

Жилец и любой публичный посетитель не должны думать «что здесь надо сделать». На каждом экране одна кнопка или одно поле.

**Нарушение:** форма с 5 полями сразу.
**Правильно:** «Введите адрес вашего дома» → кнопка → следующий экран.

В Speciālista режиме это правило ослаблено — там одна страница может содержать дашборд с несколькими действиями, потому что пользователь профессиональный.

---

### Принцип 2. Говори числами с пометкой источника

«Вы переплачиваете» — абстракция. «Расходы дома на отопление на 43% выше медианы похожих домов в районе. **Достоверность данных: средняя**» — факт с честным statusом.

**Каждое число имеет визуальный индикатор `dataConfidence`:**
- `PUBLIC_DATA` — серый кружок («публичные данные»)
- `USER_UPLOADED` — синий кружок («загружено пользователем»)
- `BOARD_VERIFIED` — зелёный галочка («подтверждено правлением»)
- `PROFESSIONAL_VERIFIED` — золотая галочка («подтверждено специалистом»)

---

### Принцип 3. Доверие строится до email-гейта

Пользователь должен убедиться, что продукт работает, прежде чем дать email. Email-гейт — обмен ценности на канал коммуникации.

**Правило:** превью (одно число) с пометкой источника достаточно убедительно, чтобы захотеть полный отчёт.

---

### Принцип 4. Мобиль — первичен

Большинство Iedzīvotāja и Valdes пользователей придут с телефона. Дизайн строится для 390px, потом адаптируется.

- Кнопки минимум 48px высотой
- Шрифт основного текста минимум 16px
- Никаких hover-эффектов как единственного сигнала
- Поля ввода — полная ширина

Speciālista режим — desktop-first (профессионалы работают за компьютером).

---

### Принцип 5. Текст на языке пользователя

LV — default, RU — второй. Все тексты переведены полностью, **никакого английского в UI**. Переключение языка — в шапке, всегда видно.

**Дополнительно:**
- Избегать юридического и технического языка для жильцов
- В Speciālista режиме можно использовать профессиональные термины (apsaimniekotājs, EIKIS, F-klase)
- Все термины проверяются по `docs/reference/readiness-glossary.md`

---

### Принцип 6. Ошибки объясняются, не обвиняют

Виновата система, не пользователь.

**Плохо:** «Ошибка 422. Файл недействителен.»
**Хорошо:** «Не удалось прочитать файл. Убедитесь, что это PDF-счёт от управляющей компании, и попробуйте ещё раз.»

---

### Принцип 7. Прогресс должен быть виден

Длинные операции (анализ PDF 5–15 сек) требуют индикатора с описанием шага. Молчащий спиннер пугает пожилого пользователя.

---

### Принцип 8 (новый). Честность важнее красоты

Если данных недостаточно — не показываем «красивое число». Показываем диапазон с честным `dataConfidence`. Если окно финансирования закрыто — пишем «Slēgts / Закрыто», а не маскируем.

**Запрещено в UI:**
- ❌ «Получите субсидию»
- ❌ «Реновация окупится»
- ❌ «Подадим вашу заявку в ALTUM»
- ❌ «Подрядчик уже готов»
- ❌ «1.5% комиссия»
- ❌ Любой английский термин

См. `readiness-glossary.md` → раздел «Формулировки, которые запрещены в UI».

---

### Цветовая философия

| Сигнал | Цвет | Когда |
|--------|------|-------|
| Аномалия / risk | Оранжевый / красный | Показываем проблему — без паники |
| Готовность / позитив | Зелёный | Высокий readiness score, открытое окно финансирования |
| Нейтральный | Серый | Данные без оценки |
| CTA (главное действие) | Синий | Одна кнопка на экране |
| Закрытое окно | Тёмно-серый | Sleg ts, недоступные сценарии |

Фон — белый. Тёмная тема — только для Speciālista режима, где есть запрос.

## Часть 3. UX-сценарии и состояния

### Модуль: Izdevumu izpratne (бывший аудит)

| Сценарий | Поведение |
|---------|-----------|
| Happy path | PDF загружен, данные распознаны → превью → email → полный отчёт + вопросы управляющему + ссылка на Readiness Score |
| Адрес не найден | «Adrese nav atrasta. Pārbaudiet pareizrakstību vai izvēlieties tuvāko.» |
| Дом есть, данных нет | Карточка с «Šajā mājā vēl neviens nav augšupielādējis rēķinu. Esiet pirmais.» |
| Файл не PDF | «Nepieciešams PDF fails. Iegūstiet to apsaimniekotāja portālā.» |
| GPT-4o не распознал | «Neizdevās nolasīt rēķinu. Iespējams, zema kvalitāte. Mēģiniet citu failu.» |
| Мало данных для бенчмарка | «Salīdzinājums būs pieejams, kad bāzē būs vairāk šīs sērijas māju.» |
| Email уже в базе | «Mēs zinām šo e-pastu. Pārskats pievienots jūsu kabinetā.» |

---

### Модуль: Финансовые сценарии

| Сценарий | Поведение |
|---------|-----------|
| Happy path | Показываем 5 сценариев со статусом окна и расчётами |
| Дом уже реновирован | «Šī māja ir atjaunota. Rādām scenārijus papildu uzlabojumiem.» |
| Дом не подходит ни под один | «Šī māja, iespējams, nav piemērota gala atbalstam — apsveriet remonta aizdevumu.» |
| Данных мало | «Provizorisks novērtējums. Lai precizētu — pieprasiet pilnu Gatavības atskaiti.» |
| ETS2 неопределённость | Disclaimer о статусе ETS2 в SCF-сценарии |

---

### Модуль: Decision Campaigns (Valdes)

| Сценарий | Поведение |
|---------|-----------|
| Happy path | Создание кампании → отправка → сбор предварительных намерений → экспорт в BIS |
| Список собственников не загружен | «Lai sāktu kampaņu, augšupielādējiet īpašnieku sarakstu (CSV)» |
| Smart-ID недоступен | «Smart-ID pagaidām nav pieejams. Mēģiniet eParaksts vai vēlāk.» |
| Юридический риск | Если legal confidence ≠ VALIDATED — кнопка «Pārbaudīt ar juristu» обязательна |
| Достигнут порог намерений | Кнопка «Eksportēt uz BIS Mājas lietu» становится главной |

---

### Модуль: Tender Room

| Сценарий | Поведение |
|---------|-----------|
| Happy path | Создание тендера → автопредложение ≥5 поставщиков → сбор предложений → сравнение → протокол |
| Связи между поставщиками | Risk indicator на UI: «Diviem dalībniekiem ir kopīgs valdes loceklis» |
| Менее 2 независимых предложений | Предупреждение: «ALTUM требует ≥2 независимых предложения. Текущая ситуация: 1.» |

---

### Шаринг (вирусный поток)

| Сценарий | Поведение |
|---------|-----------|
| Happy path | Жилец делится → сосед открывает /b/{token} → видит Readiness Score дома → загружает счёт |
| Сосед не из этого дома | «Citā mājā? [Atrodiet savu māju →]» |
| Ссылка устарела | «Saite ir novecojusi. Atrodiet māju:» + поле адреса |
| Сосед уже в базе | Email-гейт пропускается по cookie |
| Дом без данных | «Šajā mājā neviens vēl nav pārbaudījis. Esiet pirmais.» |

### Общие состояния

| Состояние | Поведение |
|-----------|-----------|
| Загрузка | Скелетон-экран |
| Ошибка сети | «Nav savienojuma. Pārbaudiet internetu.» |
| Сессия истекла | Мягкий редирект: «Bijāt neaktīvs. Pieslēdzieties vēlreiz.» |
| Пустой результат | Объяснение почему + что делать |
| Успех | Подтверждение + следующий шаг (не тупик) |

## Что меняется относительно v1

| v1 | v2 |
|----|-----|
| Один пользовательский профиль (житель + правление) | 4 режима с разной информационной плотностью |
| Главный CTA: «Начать реновацию» → голосование | Главный CTA: «Подготовить дом» → Readiness Report или Decision Campaign |
| Калькулятор ALTUM один | Финансовые сценарии (5 вариантов) |
| Email-гейт = главная цель | Email-гейт остаётся, но ведёт к trust artifact, не к маркетплейсу |
| «Переплата» как обвинительный термин | «Augstāki» / «Выше» — нейтральный |
| Голосование как UVP | Decision Campaign + экспорт в BIS |
| Шаринг без `dataConfidence` | Шаринг с пометкой «публичные данные» |
| Английские термины в UI | Только LV/RU, через `readiness-glossary.md` |
