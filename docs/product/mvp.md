# MVP — Скоуп

## Что такое MVP сейчас

Это уже не «MVP с нуля» — большая часть кода написана (см. `src/app/[locale]/`, `src/app/api/`, `prisma/schema.prisma`). MVP в новой редакции — **минимально достаточный продукт для запуска Phase 1 (Validation)** на новой концепции:

> Продать 3 платных Gatavības atskaite за €300–900 за 90 дней.

Если этого не получается — концепция платной готовности не подтверждена и нужен пивот (см. roadmap.md).

| **GDPR-консент-flow** | В MVP — упрощённый (галочка). Полный consent flow — Phase 2 |

## Минимальные требования по компонентам

### Центральный модуль

1. Prisma модель `BuildingReadinessScore` (см. data-model.md)
2. Endpoint `GET /api/readiness/:cadastralCode` — возвращает Score со всеми 8 компонентами
3. Endpoint `POST /api/readiness/:cadastralCode/recalculate`
4. UI на карточке дома: показывает score + nextBestAction
5. Логика генерации `nextBestAction` — детерминированная функция в `src/lib/readiness/`

### Финансовые сценарии

1. Конфиг файлы в `src/lib/financing/rules/` — по одному на сценарий
2. Endpoint `GET /api/financing/scenarios?cadastralCode=...` — возвращает массив 5 сценариев с расчётами
3. UI: страница `/renovation/scenarios` (или `/financing`) — таблица сценариев со статусом окна и оценкой
4. Локализация всех текстов через `messages/lv.json` и `messages/ru.json` под новые термины

### Decision Campaigns

1. Расширить `VotingCampaign` модель: добавить `decisionType: DecisionType`
2. UI: список 7 типов решений на странице `/voting/new` с объяснением каждого
3. Шаблоны вопросов на LV/RU для каждого типа
4. PDF-экспорт пакета решения для BIS

### Аудит (перепозиционирование)

1. На странице `/audit/report/<id>` добавить:
   - Раздел «**Jautājumi apsaimniekotājam**» / «Вопросы для управляющего»
   - Раздел «**Nākamais solis**» / «Следующий шаг» — ссылка на дом и Readiness Score
2. Удалить со всех страниц текст «получите субсидию ALTUM», «реновация окупится за X лет» (категоричные утверждения)

### Платный Gatavības atskaite

1. Страница `/readiness-report/order` — форма
2. Stripe / payment provider integration (отдельно от подписок)
3. Backend job: генерация PDF (template-based)
4. Email с attachment + ссылкой в личный кабинет
5. Запись в БД: `ReadinessReportOrder` (новая модель)

## Критерии выхода из Phase 1 → Phase 2

1. **3+ оплаченных Gatavības atskaite** — платёжная готовность подтверждена
2. **10+ интервью** с biedrības / mājas vecākais — мы знаем настоящую боль
3. **1 пилот** с biedrība в режиме Valdes — есть рабочий референс
4. **0 публичных обещаний** субсидии или success fee
5. **Все 8 компонентов Readiness Score реализованы** на minimum уровне (хотя бы заглушки с честным `data_confidence`)

## Риски MVP

| Риск | Вероятность | Mitigation |
|------|-------------|-----------|
| **Никто не платит €300+ за отчёт** | Средняя | После 90 дней — пивот в чистый B2B SaaS Speciālista, отказ от B2C-входа |
| **Rules engine упрощённый, ошибается в edge cases** | Высокая | На UI явный disclaimer «provizorisks novērtējums», `data_confidence` маркеры, ссылка «проверьте у юриста» |
| **GPT-4o парсит счёт неточно** | Средняя | `parseConfidence: LOW` помечает ручной проверкой, fallback на полу-ручной ввод |
| **biedrība не загружает CSV собственников** | Высокая | Шаблон в платформе, видеоинструкция, шаблон в Excel/CSV |
| **Smart-ID/eParaksts integration сломается** | Низкая | Уже работает в коде, мониторинг |
| **Хардкод 1.5% commission в коде попадёт на UI** | Средняя | После rewrite доков — отдельная итерация по коду: удалить из `api/tenders/*` |
