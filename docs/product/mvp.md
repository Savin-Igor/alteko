# MVP — Скоуп и маркетплейс

## Часть 1. MVP

### Что такое MVP ALTEKO

MVP — полный продукт без маркетплейса. Цель: доказать, что пользователь проходит весь путь от загрузки счёта до протокола голосования. Прямой выручки нет — первые сделки закрываются вручную.

---

### Модули MVP

| Модуль | Входит | Не входит |
|--------|--------|-----------|
| **Аудит** | Поиск адреса, карточка дома, загрузка PDF, OCR+GPT-4o, бенчмарк по серии/району/площади, email-гейт, полный отчёт | — |
| **Реновация** | Прогноз экономии (сегментный — без email, персональный — после email), калькулятор субсидии Altum, генерация документов | — |
| **Голосование** | Smart-ID / eParaksts, кампания опроса, учёт голосов, протокол ≥50% | — |
| **B2B** | Дашборд для правления biedrības: история счетов, бенчмарк дома, отчёты для собрания | Управляющие компании |
| **Маркетплейс** | — | Весь (Фаза 2) |

---

### Минимальные требования по модулям

#### Аудит
- Поиск адреса с автодополнением (Jāņa sēta → LVM WFS → PostgreSQL)
- Загрузка PDF-счёта (S3)
- Парсинг одним вызовом GPT-4o vision
- Бенчмаркинг: минимум P25/P50/P75 по сегменту
- Email-гейт: превью (одно число отклонения) без email, полный отчёт после email
- Отчёт по email + в личном кабинете

#### Реновация
- Средняя экономия по сегменту — без email (превью)
- Персональный расчёт (€/мес., субсидия Altum, срок окупаемости) — после email
- Генерация двух документов: заявление о намерении и повестка собрания

#### Голосование
- Список собственников — ручной CSV от правления (Zemesgrāmata API нет)
- Аутентификация через Smart-ID или eParaksts
- Голосование: «за» / «против» / «воздержался»
- Автоматический подсчёт: достигнут ли порог ≥50%
- Генерация протокола опроса (принимается Altum)

#### B2B (правление biedrības)
- Дашборд дома: история расходов, бенчмарк, аномалии
- Экспорт отчёта для собрания жильцов (PDF)
- Управление голосованием: создание кампании, загрузка списка собственников

---

### Что не строим в MVP

| Исключение | Причина |
|-----------|---------|
| Маркетплейс подрядчиков | Нет объектов для тендера на старте |
| Управляющие компании (B2B tier 2) | Конфликтный сегмент, требует отдельного позиционирования |
| Автоматический мониторинг счетов | Нужна критическая масса данных сначала |
| Мобильное приложение | Web SSR достаточно для MVP |
| Интеграция с Zemesgrāmata | API нет, ручной CSV — достаточно |

---

### Критерии выхода из MVP (когда запускать Фазу 2)

1. **10 домов** прошли голосование с результатом ≥50%
2. **500 домов** в базе с загруженными счетами (бенчмаркинг точный)
3. **3–5 подрядчиков** выразили готовность платить комиссию за лиды
4. Ручное закрытие сделок стало узким местом (>2 объекта одновременно)

Когда все четыре условия выполнены — строим маркетплейс.

---

### Риски MVP

| Риск | Вероятность | Mitigation |
|------|-------------|-----------|
| Пользователи не загружают PDF | Высокая | A/B тест: ручной ввод vs PDF |
| Бенчмарк неточный (мало данных) | Высокая | Запустить с ≥50 домов до публичного релиза |
| Голосование — долгий цикл (месяцы) | Высокая | Параллельно закрывать первые сделки вручную |
| Подрядчики не придут без объектов | Высокая | Первые 3 подрядчика — личные переговоры |
| Smart-ID интеграция сложнее ожидаемого | Средняя | Заложить 2–3 недели буфера |

## Часть 2. Идея маркетплейса

### Что такое маркетплейс ALTEKO

Маркетплейс — это не биржа объявлений. Это **конвейер предквалифицированных лидов для подрядчиков**.

Разница принципиальная:

| Обычная биржа | Маркетплейс ALTEKO |
|--------------|-------------------|
| Подрядчик сам ищет объекты | Объект приходит с готовым протоколом ≥50% |
| Нет информации о готовности дома | Известны: серия, год, площадь, энергокласс, бюджет Altum |
| Нужно убеждать жителей с нуля | Жители уже проголосовали — решение принято |
| Конкуренция по цене за холодный лид | Конкуренция за горячий лид с документами |

---

### Ценность для подрядчика

Обычный путь подрядчика к объекту реновации:
1. Маркетинг → 2. Холодный контакт с правлением → 3. Убеждение жителей → 4. Сбор голосов → 5. Оформление документов → 6. Подача в Altum → 7. Контракт

Это 6–18 месяцев работы до контракта.

ALTEKO передаёт объект на шаге 6–7: протокол голосования уже есть, документы сгенерированы, бюджет рассчитан. Подрядчик приходит только на торги.

**Поэтому подрядчик готов платить комиссию 1–2%** — это дешевле, чем строить весь путь самому.

---

### Механика тендера

```
Дом получил протокол ≥50%
        ↓
ALTEKO создаёт тендер автоматически
        ↓
Верифицированным подрядчикам приходит уведомление
        ↓
Подрядчики подают предложения (цена + срок + условия)
        ↓
Правление дома выбирает подрядчика
        ↓
Контракт подписан → ALTEKO получает комиссию
```

---

### Верификация подрядчиков

Не каждый может участвовать в тендере. Условия:

- Регистрация в Lursoft (действующее юрлицо)
- Лицензия строительного подрядчика (BVKB реестр)
- Минимум 2 завершённых объекта реновации с отзывами
- Подписанное соглашение о комиссии с ALTEKO

Рейтинг строится на: сроки, бюджет, отзывы жителей, процент объектов с Altum-одобрением.

---

### Почему маркетплейс — Фаза 2, а не MVP

Маркетплейс без объектов — пустая витрина. Подрядчики не зарегистрируются, если нет тендеров. Тендеры не появятся без домов, прошедших голосование. Голосование занимает месяцы.

**Порядок правильный:**
1. MVP собирает данные и проводит голосования
2. Первые 10 объектов закрываются вручную (личные переговоры с подрядчиками)
3. На этих 10 объектах обкатывается процесс и собираются отзывы
4. Маркетплейс запускается как автоматизация уже работающего ручного процесса

Это классический путь: делать вручную то, что потом автоматизируешь.
