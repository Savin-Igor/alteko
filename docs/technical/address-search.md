# Модуль поиска по адресу

## Кратко

Пользователь вводит адрес → автодополнение → выбирает здание → backend определяет кадастровое обозначение → загружает данные здания из локальной базы.

Один API не даёт всё необходимое. Цепочка: поиск адреса (Jāņa sēta) → координаты → пространственный WFS-запрос (LVM GeoServer) → локальная база зданий (открытые данные VZD + энергетические сертификаты BVKB).

---

## Источники данных (верифицированы)

| Назначение | Источник | Тип | Авторизация |
|-----------|---------|-----|------------|
| Автодополнение адреса + геокодинг | API Jāņa sēta (developers.kartes.lv) | Live API | API-ключ (есть бесплатный тариф) |
| Координаты → кадастровое обозначение | LVM GeoServer WFS (lvmgeoserver.lvm.lv) | Live API | Нет — бесплатный публичный |
| Атрибуты здания (площадь, год, материал, этажи) | Открытые данные VZD Building.ZIP (data.gov.lv) | Локальная БД | Нет — CC BY |
| Класс энергоэффективности (A–G) | Энергетические сертификаты BVKB (data.gov.lv) | Локальная БД | Нет — CC0 |
| Тайлы карт | OpenStreetMap / Mapbox | Live API | Нет / API-ключ |

**Чего нет в открытых данных:** Серия здания (103, 119, 467 и т.д.) — не является структурированным полем ни в одном открытом наборе. Должна выводиться эвристически (год + материал стен + количество этажей) или заноситься вручную в локальную БД.

## Архитектура

```
ПОЛЬЗОВАТЕЛЬ
  │
  │ вводит "Brīvības 55"
  ▼
СТРАНИЦА NEXT.JS (React)
  │
  ├─ Поле автодополнения
  │    GET /api/address/search?q=Brīvības+55
  │    показывает: выпадающий список с 5–8 подсказками
  │
  │ пользователь выбирает адрес
  ▼
API ROUTE NEXT.JS: /api/address/resolve
  │
  ├─ 1. Вызов API геокодинга Jāņa sēta
  │       → возвращает: lat, lon, vzd_id (официальный код адреса)
  │
  ├─ 2. Вызов LVM GeoServer WFS (BBOX вокруг lat/lon, радиус ~10м)
  │       слой: publicwfs:kkbuilding
  │       → возвращает: cadastralCode (kadastra apzīmējums)
  │
  ├─ 3. Запрос к PostgreSQL через Prisma
  │       prisma.building.findUnique({ where: { cadastralCode } })
  │       → возвращает: areaM2, constructionYear, energyClass,
  │                     series, lat, lon
  │
  └─ 4. Возвращает JSON на frontend:
          id, address, lat, lon,
          series, constructionYear, areaM2, energyClass,
          cadastralCode
  │
  ▼
СТРАНИЦА NEXT.JS (React)
  ├─ Показывает карточку здания с ключевыми атрибутами
  ├─ Выделяет здание на карте
  └─ Активирует кнопки «Анализировать расходы» / «Рассчитать реновацию»
```

## Локальная база данных зданий

Два открытых набора данных импортируются в PostgreSQL и обновляются автоматически.

### Таблица: `building_cache`

```sql
CREATE TABLE building_cache (
    kadastrs_code       varchar(14) PRIMARY KEY,  -- kadastra apzīmējums
    vzd_id              varchar(9),               -- код адреса ARVIS (VARISCode)
    address             text,
    area_m2             decimal,
    year_built          integer,
    floor_count         integer,
    wall_material_code  varchar(10),
    energy_class        char(1),                  -- A–G, из BVKB
    series              varchar(10),              -- производное, nullable
    lat                 decimal,
    lon                 decimal,
    vzd_updated_at      date,
    bvkb_updated_at     date
);
```

### Расписание обновления данных

| Набор данных | Источник | Частота | Объём |
|-------------|---------|---------|-------|
| Building.ZIP | data.gov.lv (VZD) | Еженедельно | ~23 500 зданий |
| Энергетические сертификаты | data.gov.lv (BVKB) | Ежедневно | ~23 500 записей |

Оба набора достаточно малы для полной перезагрузки при каждом обновлении. Фоновый скрипт (`scripts/sync-buildings.ts`, запускается через cron или по расписанию GitHub Actions) обрабатывает скачивание и upsert через Prisma.

### Вывод серии здания

Серия отсутствует в любом официальном наборе данных.

**Реализация:** `scripts/assign-series.ts` — автономный скрипт с 11 правилами для серий (Stalinka, Khrushchevka, 103, 104, 119, 316, 318, 464, 467, 602, 602P). Запуск: `make assign-series`. Обновляет поле `Building.series` в базе через Prisma.

**Принцип:** каждому зданию сопоставляется первое совпавшее правило по комбинации `(yearFrom–yearTo) + wallMaterial + (floorsMin–floorsMax)`. Пакетное обновление по 500 записей.

**Ограничения:**
- Серии 104, 119, 602 сильно перекрываются по году/этажам → результат «best-effort»
- `Dzelzsbetona paneļi` в VZD охватывает все ж/б панели, подтипы не различимы
- Серии 316/318 в скрипте сопоставлены с `wallMaterial = 'Dzelzsbetona'` — возможное расхождение с реальной классификацией VZD (по источникам, 316/318 — силикатный кирпич)

Поле `Building.series` хранится как `nullable`. Ручные исправления могут вносить администраторы.

**Исследование по сериям:** подробные данные по каждой серии (конструкция, энергопотребление, нормативный срок службы, типичные проблемы) → `docs/research/building-series.md`

## Frontend: автодополнение

```
Пользователь вводит →
  debounce 250 мс →
    GET /api/address/search?q={input} →
      backend вызывает API поиска адресов Jāņa sēta →
        возвращает 5–8 подсказок →
          рендерится выпадающий список
```

Каждая подсказка показывает: улица + номер дома + город + почтовый индекс.

При выборе: один вызов `GET /api/address/resolve?vzd_id={id}` возвращает полную запись о здании (шаги 2–4 выше). Карта перемещается к координатам и выделяет контур здания.

**Обработка языка:**
- Jāņa sēta обрабатывает латышский, русский и смешанный ввод
- «Brīvības» и «Брīвīбас» разрешаются в один и тот же адрес
- UI-поле принимает любой язык — backend нормализует

## Карта

**Провайдер:** тайлы OpenStreetMap через Leaflet.js (бесплатно, API-ключ не нужен).

**Опционально:** Mapbox для рендеринга более высокого качества (платно, ~$0,50/1000 запросов тайлов).

Контур здания на карте берётся из WMS-слоя LVM GeoServer (`publicwfs:kkbuilding`) — того же слоя, что используется для кадастрального поиска. Это означает, что выделенный полигон — это реальная кадастровая граница здания, а не геокодированная точка.

```
После разрешения здания:
  frontend загружает WMS-тайл с BBOX, отфильтрованным по kadastra_apzimejums здания
  → реальный контур здания рендерится как полигон на карте
```

## Кеширование

| Что | Кеш | TTL |
|-----|-----|-----|
| Результат геокодинга Jāņa sēta (адрес → lat/lon + vzd_id) | Таблица PostgreSQL `geocode_cache` | 30 дней |
| Результат LVM WFS (lat/lon → kadastrs_code) | PostgreSQL (хранится в записи здания) | До еженедельного обновления |
| Атрибуты здания | Локальная БД (building_cache) | Обновляется еженедельно |
| Подсказки автодополнения | In-memory (Redis или кеш Next.js) | 1 час |

Результаты геокодинга стабильны — адреса в Латвии редко меняют координаты. Кеширование избавляет от повторных API-вызовов для зданий, которые уже искались.

## Оценка стоимости API

| Сервис | Предполагаемый объём | Стоимость |
|--------|---------------------|----------|
| Геокодинг Jāņa sēta | 1 000 уникальных адресов/мес. | Бесплатный тариф; уточнить тарифы на kartes@kartes.lv |
| LVM GeoServer WFS | 1 000 запросов/мес. | Бесплатно, лимиты не задокументированы |
| OSM тайлы (Leaflet) | Без ограничений | Бесплатно (политика использования применяется) |
| Mapbox тайлы | 50 000 загрузок тайлов/мес. | Бесплатный тариф |

## План внедрения

**Фаза 1 (MVP — необходимо для модуля аудита):**
1. Получить API-ключ Jāņa sēta
2. Собрать эндпоинт `/api/address/search` (прокси к автодополнению Jāņa sēta)
3. Собрать эндпоинт `/api/address/resolve` (геокодинг → WFS → поиск в локальной БД)
4. Импортировать VZD Building.ZIP + энергетические сертификаты BVKB в `building_cache`
5. Настроить задание еженедельного/ежедневного обновления
6. Собрать компонент автодополнения на frontend (карта Leaflet опциональна в MVP)

**Фаза 2 (полный UX):**
7. Добавить карту Leaflet с контуром здания из LVM WMS
8. Добавить эвристику вывода серии здания + UI исправлений для администраторов
9. Кешировать результаты геокодинга в `geocode_cache`
10. Добавить обработку ввода адреса на русском языке

**Фаза 3 (качество данных):**
11. Краудсорсинг исправлений серий от пользователей («Это ваша серия дома?»)
12. Оценить контрактный веб-сервис VZD для более богатых данных, если эвристики окажутся недостаточными

## Открытые вопросы

1. **Тарифы Jāņa sēta** — связаться с kartes@kartes.lv для получения объёмных тарифов до запуска в продакшн
2. **Точность серии здания** — эвристический подход требует валидации на выборке; рассмотреть получение обогащённого набора данных от провайдера данных о недвижимости
3. **Лимиты LVM GeoServer** — лимиты не задокументированы; протестировать под нагрузкой до запуска
4. **Поддержка сельских адресов** — база адресов VZD полна для сельской местности, но покрытие Jāņa sēta требует верификации за пределами Риги
