# Внешние интеграции

## Обзор

ALTEKO интегрируется с шестью категориями внешних систем. Здесь описаны назначение, передаваемые данные и известные ограничения для каждой из них.

Подробности по источникам данных (владение, тип доступа) — в [data-sources.md](../reference/data-sources.md).

---

## 1. Kadastrs.lv (Valsts zemes dienests)

**Назначение:** Получение метаданных здания при вводе адреса пользователем.

**Получаемые данные:**
- Серия здания
- Год постройки
- Общая площадь (м²)
- Количество квартир
- Класс энергоэффективности (A–G)
- Кадастровый номер (используется как стабильный идентификатор здания)

**Подход к интеграции:**
- **Live API не нужен.** VZD публикует открытые данные `Building.ZIP` (еженедельно, CC BY) на data.gov.lv. Этот набор содержит все здания с площадью, годом постройки, материалом стен, количеством этажей и VARISCode (код адреса).
- Класс энергоэффективности (A–G) берётся из набора данных BVKB об энергетических сертификатах (ежедневно, CC0), связывается через `cadastralCode`.
- Фоновый скрипт (`scripts/sync-buildings.ts`) еженедельно скачивает оба набора и делает upsert в таблицу `Building` через Prisma.
- Контрактный веб-сервис VZD (более богатые данные в реальном времени): требует формального соглашения с Корпоративным центром VZD — оценить после MVP.

**Триггер:** Еженедельный cron → скрипт синхронизации → upsert через Prisma → таблица `Building` обновлена. Поиск адреса читает из локальной БД, а не из live-Kadastrs.

## 2. Zemesgrāmata (Земельный реестр)

**Назначение:** Получение списка собственников квартир в доме для:
- Определения круга участников голосования
- Расчёта весов долей собственности при голосовании
- Верификации того, что авторизованный пользователь действительно является владельцем заявленной квартиры

**Получаемые данные:**
- Список квартир с именами текущих собственников
- Доля собственности (дробь от общей площади дома)
- Обременения (информационно)

**Подход к интеграции:**
- **Публичного API нет.** Zemesgrāmata работает только через веб-портал с оплатой за каждый запрос (~€5/запрос). Никакого пакетного REST API через zemesgramata.lv, data.gov.lv или порталы разработчиков latvija.lv не существует.
- **Варианты обходного пути:**
  1. Ручной импорт: правление товарищества скачивает выписку о собственниках с портала и загружает CSV в ALTEKO
  2. Нотариальный путь: нотариус получает заверенный список собственников (законно, но добавляет трудозатраты и стоимость)
  3. Самодекларация: пользователи сами заявляют свою долю собственности, ALTEKO хранит её без верификации (допустимо для аудита в Фазе 1; недостаточно для юридически значимого голосования)
- Для Фазы 3 (голосование) Вариант 1 (ручная загрузка CSV администратором товарищества) — наиболее практичный путь без API
- Правовое замечание: обработка персональных данных о собственности регулируется GDPR

**Триггер:** Создание VotingCampaign → правление загружает CSV → заполняются записи Apartment с данными о собственниках и долях.

## 3. Altum.lv

**Назначение:** Предоставление актуальных условий субсидийной программы для калькулятора реновации.

**Получаемые данные:**
- Действующие программы (названия, типы допустимых зданий, проценты покрытия)
- Максимальная допустимая стоимость реновации на м²
- Контрольный список требований к заявке

**Подход к интеграции:**
- Публичного API не подтверждено
- Данные поддерживаются вручную, обновляются при изменении условий программ Altum
- Для MVP: статические конфигурационные файлы с параметрами программы
- Для Фазы 3: скрейпер или обнаружение изменений через RSS

**Триггер:** Загрузка калькулятора реновации → backend читает текущий конфиг программы → рассчитывается сумма субсидии.

## 4. Lursoft

**Назначение:** Верификация финансового состояния и регистрационного статуса подрядчика перед включением в маркетплейс.

**Получаемые данные:**
- Статус регистрации компании (активна / ликвидирована / приостановлена)
- Отсутствие активного производства о несостоятельности
- Годовая выручка (для оценки размера)
- Структура собственности

**Подход к интеграции:**
- Платный REST API — требует коммерческого соглашения с Lursoft
- Вызывается однократно при верификации заявки подрядчика
- Периодическая повторная верификация (ежеквартально) для отслеживания изменений статуса

**Триггер:** Подрядчик подаёт заявку на онбординг → backend вызывает API Lursoft → результат верификации сохраняется в записи Contractor.

## 5. Smart-ID

**Назначение:** Аутентификация владельцев квартир для юридически значимого электронного голосования.

**Подход к интеграции:**
- API Smart-ID (SK ID Solutions) — вызывается из API Route Next.js (`/api/voting/smart-id`)
- Поток:
  1. Пользователь инициирует голосование → API route вызывает эндпоинт `authenticate` Smart-ID с персональным кодом пользователя
  2. Smart-ID отправляет вызов на мобильное приложение пользователя
  3. Пользователь подтверждает → API route получает подписанный ответ с сертификатом
  4. Подпись сохраняется в записи `Vote` в PostgreSQL через Prisma
- Требует соглашения с SK ID Solutions

**Замечание по безопасности:** Персональный код — чувствительные персональные данные (специальная категория по GDPR). Шифруется перед сохранением в поле `User.smartIdCode`. Никогда не логируется.

## 6. eParaksts

**Назначение:** Альтернативный метод аутентификации для голосования — для пользователей без Smart-ID.

**Подход к интеграции:**
- Латвийская государственная система электронной подписи, оператор — VAS «Latvijas Valsts radio un televīzijas centrs» (LVRTC)
- OAuth 2.0-поток — обрабатывается в API Route Next.js (`/api/voting/eparaksts`)
- eParaksts mobile: аналогично Smart-ID (подтверждение через приложение)
- eParaksts USB: браузерный поток с аппаратным токеном

**Триггер:** Пользователь выбирает «eParaksts» на экране голосования → OAuth redirect → callback-route Next.js → подпись сохраняется в записи `Vote` через Prisma.

## 7. Weather API

**Назначение:** Предоставление исторических данных о градусо-днях отопления (HDD) для тепловых прогнозов.

**Рекомендуемый провайдер:** Open-Meteo (open-source, бесплатно для некоммерческого использования)
**Получаемые данные:**
- Исторические дневные температуры по координатам
- Градусо-дни отопления, вычисляемые из сырой температуры

**Подход к интеграции:**
- REST API, аутентификация не требуется (бесплатный тариф)
- Вызывается из API Route Next.js (`/api/renovation/calculate`) при генерации прогноза экономии
- Результат кешируется в записи `Building` в PostgreSQL — повторная загрузка не нужна, если здание или период не изменились
