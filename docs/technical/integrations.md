# Внешние интеграции

## Обзор

ALTEKO интегрируется с десятью категориями внешних систем. Здесь описаны назначение, передаваемые данные и известные ограничения для каждой из них.

Подробности по источникам данных (владение, тип доступа) — в [data-sources.md](../reference/data-sources.md).

---

## 1. Kadastrs.lv (Valsts zemes dienests)

**Назначение:** Получение метаданных здания при вводе адреса пользователем.

**Получаемые данные:**
- Серия здания
- Год постройки
- Общая площадь (м²)
- Количество квартир
- Класс энергоэффективности (A–G)
- Кадастровый номер (используется как стабильный идентификатор здания)

**Подход к интеграции:**
- **Live API не нужен.** VZD публикует открытые данные `Building.ZIP` (еженедельно, CC BY) на data.gov.lv. Этот набор содержит все здания с площадью, годом постройки, материалом стен, количеством этажей и VARISCode (код адреса).
- Класс энергоэффективности (A–G) берётся из набора данных BVKB об энергетических сертификатах (ежедневно, CC0), связывается через `cadastralCode`.
- Фоновый скрипт (`scripts/sync-buildings.ts`) еженедельно скачивает оба набора и делает upsert в таблицу `Building` через Prisma.
- Контрактный веб-сервис VZD (более богатые данные в реальном времени): требует формального соглашения с Корпоративным центром VZD — оценить после MVP.

**Триггер:** Еженедельный cron → скрипт синхронизации → upsert через Prisma → таблица `Building` обновлена. Поиск адреса читает из локальной БД, а не из live-Kadastrs.

## 2. Zemesgrāmata (Земельный реестр)

**Назначение:** Получение списка собственников квартир в доме для:
- Определения круга участников голосования
- Расчёта весов долей собственности при голосовании
- Верификации того, что авторизованный пользователь действительно является владельцем заявленной квартиры

**Получаемые данные:**
- Список квартир с именами текущих собственников
- Доля собственности (дробь от общей площади дома)
- Обременения (информационно)

**Подход к интеграции:**
- **Публичного API нет.** Zemesgrāmata работает только через веб-портал с оплатой за каждый запрос (~€5/запрос). Никакого пакетного REST API через zemesgramata.lv, data.gov.lv или порталы разработчиков latvija.lv не существует.
- **Варианты обходного пути:**
  1. Ручной импорт: правление товарищества скачивает выписку о собственниках с портала и загружает CSV в ALTEKO
  2. Нотариальный путь: нотариус получает заверенный список собственников (законно, но добавляет трудозатраты и стоимость)
  3. Самодекларация: пользователи сами заявляют свою долю собственности, ALTEKO хранит её без верификации (допустимо для аудита в Фазе 1; недостаточно для юридически значимого голосования)
- Для Фазы 3 (голосование) Вариант 1 (ручная загрузка CSV администратором товарищества) — наиболее практичный путь без API
- Правовое замечание: обработка персональных данных о собственности регулируется GDPR

**Триггер:** Создание VotingCampaign → правление загружает CSV → заполняются записи Apartment с данными о собственниках и долях.

## 3. Altum.lv

**Назначение:** Источник актуальных условий программ ALTUM для финансовых сценариев.

**Получаемые данные:**
- Статус программ ALTUM (открыта/закрыта/ожидается):
  - **ALTUM 2021-2027** — закрыта для новых заявок (с 13.05.2025), все €173M зарезервированы под 338 заявок
  - **ALTUM remonta aizdevums** — открыта (€10k+, 3.9%, до 30.06.2031)
- Максимальная допустимая стоимость реновации на м²
- Контрольный список требований к заявке (включая правила выбора поставщиков: ≥5 опрошенных, ≥2 независимых)

**Подход к интеграции:**
- Публичного API нет
- Данные поддерживаются вручную в `src/lib/financing/rules/altum-*.ts` и обновляются при изменении условий
- Для Phase 3: возможен скрейпер или обнаружение изменений через RSS
- API mans.altum.lv (личный кабинет) — публичного нет; экспорт данных в виде PDF-пакета для ручной подачи

**Триггер:** Расчёт финансовых сценариев → backend читает текущую версию rules engine → возвращает пригодность.

## 3a. Sociālā klimata fonds (SCF) — будущее окно

**Назначение:** Источник правил для основного будущего источника финансирования (€127M на МКД).

**Источники данных:**
- **likumi.lv/ta/id/361681** — Latvijas SCF plāns 2026-2032
- **tapportals.mk.gov.lv** — TAP портал Кабмина (отслеживание изменений)
- **employment-social-affairs.ec.europa.eu** — статус одобрения плана Еврокомиссией

**Текущий статус (май 2026):**
- План принят MK rīkojums Nr.393 (2.07.2025)
- На оценке Еврокомиссии (одобрена пока только Швеция)
- MK noteikumi для меры МКД ожидаются в Q4 2026
- Приём заявок до Q2 2027

**Подход к интеграции:**
- В Phase 1: статичные правила в `src/lib/financing/rules/scf-2026-2032.ts` на основе текста плана с пометкой `confidence: low`
- В Phase 3: после выхода MK noteikumi — обновление правил, новая версия в `RulesEngineVersion`, batch-пересчёт `BuildingReadinessScore` для всех зданий
- Подача заявок через KPVIS (Kohēzijas politikas fondu vadības informācijas sistēma) — API публичного нет, ALTEKO готовит PDF-пакет

## 3b. KPVIS (Kohēzijas politikas fondu vadības informācijas sistēma)

**Назначение:** Будущая официальная система подачи проектных заявок SCF.

**Подход к интеграции:**

## 4. Lursoft

**Назначение:**
1. Верификация финансового состояния и регистрационного статуса подрядчика перед включением в Tender Room
2. **Supplier Risk Check** (новое в v2): построение графа связей между участниками тендера для предотвращения конфликта интересов

**Получаемые данные:**
- Статус регистрации компании (активна / ликвидирована / приостановлена)
- Отсутствие активного производства о несостоятельности
- Годовая выручка
- Структура собственности (правление, владельцы, patiesā labuma guvēji)
- Адрес регистрации

**Подход к интеграции:**
- Платный REST API — требует коммерческого соглашения с Lursoft
- Вызывается:
  - При верификации заявки подрядчика (однократно)
  - При создании тендера (для всех участников — построение графа связей)
  - Ежеквартально для повторной верификации

> **Статус реализации:** В текущем коде (`api/contractors/verify`) Lursoft-вызов замоканнен для dev-режима. Production API-ключ необходим перед запуском Phase 2 (Tender Room с Risk Check).

**Триггер:**
- Подрядчик подаёт заявку → backend вызывает API Lursoft → результат сохраняется в `Contractor`
- Создание тендера → backend проверяет всех участников → indicators в UI

## 4a. IUB open data (Iepirkumu uzraudzības birojs)

**Назначение (новое в v2):** Дополнить Supplier Risk Check данными о публичных тендерах, жалобах, дисквалификациях.

**Получаемые данные:**
- Procurement notices (тендеры)
- Заключённые контракты, изменения договоров
- Жалобы и расследования
- Дисквалификации

**Подход к интеграции:**
- Open data в JSON/XML, обновляется ежедневно
- Скрипт `scripts/sync-iub.ts` загружает данные, строит локальный граф связей
- Используется в `src/lib/supplier-risk/`

**Источник:** iub.gov.lv/en/open-data

## 5. Smart-ID

**Назначение:** Аутентификация владельцев квартир для юридически значимого электронного голосования.

**Подход к интеграции:**
- API Smart-ID (SK ID Solutions) — вызывается из API Route Next.js (`/api/voting/smart-id`)
- Поток:
  1. Пользователь инициирует голосование → API route вызывает эндпоинт `authenticate` Smart-ID с персональным кодом пользователя
  2. Smart-ID отправляет вызов на мобильное приложение пользователя
  3. Пользователь подтверждает → API route получает подписанный ответ с сертификатом
  4. Подпись сохраняется в записи `Vote` в PostgreSQL через Prisma
- Требует соглашения с SK ID Solutions

**Замечание по безопасности:** Персональный код — чувствительные персональные данные (специальная категория по GDPR). Шифруется перед сохранением в поле `User.smartIdCode`. Никогда не логируется.

## 6. eParaksts

**Назначение:** Альтернативный метод аутентификации для голосования — для пользователей без Smart-ID.

**Подход к интеграции:**
- Латвийская государственная система электронной подписи, оператор — VAS «Latvijas Valsts radio un televīzijas centrs» (LVRTC)
- OAuth 2.0-поток — обрабатывается в API Route Next.js (`/api/voting/eparaksts`)
- eParaksts mobile: аналогично Smart-ID (подтверждение через приложение)
- eParaksts USB: браузерный поток с аппаратным токеном

**Триггер:** Пользователь выбирает «eParaksts» на экране голосования → OAuth redirect → callback-route Next.js → подпись сохраняется в записи `Vote` через Prisma.

## 7. Weather API

**Назначение:** Предоставление исторических данных о градусо-днях отопления (HDD) для тепловых прогнозов.

**Рекомендуемый провайдер:** Open-Meteo (open-source, бесплатно для некоммерческого использования)

**Подход к интеграции:**
- REST API, аутентификация не требуется
- Вызывается из API Route Next.js при генерации прогноза экономии
- Результат кешируется в `Building`

> **Статус:** В текущем коде используются только коэффициенты по энергоклассу. Weather API — улучшение Phase 2.

## 8. BIS Mājas lieta (новое в v2)

**Назначение:** Экспорт решений собственников и протоколов в государственный электронный «дело дома».

**Что уже умеет BIS:**
- Электронные опросы собственников
- Электронные собрания
- Голосование с мобильного через Smart-ID/eParaksts
- Хранение протоколов в Mājas lieta

**Подход к интеграции:**
- **API публичного нет** на май 2026
- Phase 1: ALTEKO генерирует PDF-пакет (повестка, шаблон протокола, пакет согласий) для ручной выгрузки уполномоченным лицом в BIS
- Phase 3: следить за публикацией API BIS; при появлении — добавить прямой экспорт

**Источник:** bis.gov.lv/par-mums/buvniecibas-informacijas-sistema/majas-lieta

> **Принцип:** ALTEKO **не дублирует** BIS. BIS — государственный контур; ALTEKO — подготовительный слой до BIS.
