# Технический стек

## Язык

**TypeScript** — используется во всей кодовой базе: frontend, backend API, скрипты данных, тесты.

---

## Frontend + Backend

**Фреймворк:** Next.js (React)
**Рендеринг:** Server-Side Rendering (SSR) + API Routes

Next.js выполняет роль как frontend, так и backend в одном приложении:
- **Pages / App Router** — SSR-страницы для всех пользовательских экранов (поиск адреса, дашборды, калькулятор реновации, голосование)
- **API Routes (`/api/*`)** — backend-эндпоинты: обработка документов, бенчмаркинг, геокодинг, LLM-вызовы, аутентификация Smart-ID, логика голосования

Это устраняет необходимость в отдельном backend-сервере в MVP. Один Docker-контейнер запускает всё приложение.

SSR предпочтителен по следующим причинам:
- SEO важен для органического привлечения (пользователи ищут по адресу)
- Страницы с большим объёмом данных выигрывают от серверной загрузки данных
- Нет сложностей с CORS между frontend и API

## ORM и база данных

**ORM:** Prisma
**База данных:** PostgreSQL

Prisma — единственный источник правды для схемы базы данных. Файл `schema.prisma` определяет все модели — никаких отдельных файлов миграций, никакого ручного SQL.

```
prisma/
├── schema.prisma        ← определения моделей
└── migrations/          ← автогенерируется через prisma migrate dev
```

Полная Prisma schema — в [data-model.md](data-model.md).

## AI / LLM

**Подход:** API-first. Нет локальной модели, нет отдельного AI-сервиса.

Все LLM-вызовы делаются из API Routes Next.js с использованием OpenAI SDK (TypeScript):
- Парсинг документов: PDF → base64 → GPT-4o vision → структурированный JSON
- Системный и пользовательский промпт инлайн в `src/lib/llm.ts`

```ts
// Пример: route парсинга расходов
import OpenAI from 'openai'
const client = new OpenAI()
const result = await client.chat.completions.create({
  model: 'gpt-4o',
  messages: [{ role: 'user', content: [{ type: 'image_url', ... }] }],
  response_format: { type: 'json_object' }
})
```

**Провайдер по умолчанию:** GPT-4o. LLM-клиент абстрагирован в `src/lib/llm.ts`, поэтому провайдера можно заменить, не трогая код функций.

**Тепловой прогноз и обнаружение аномалий:** детерминированные TypeScript-функции, LLM не нужен. См. `src/lib/calculator/` для тепловой модели и `src/lib/benchmarks/` для обнаружения аномалий.

## Файловое хранилище

**S3-совместимое объектное хранилище** (AWS S3 или любой совместимый провайдер).

Используется для:
- Загружаемых PDF с расходами
- Сгенерированных документов (протоколы собраний, заявки на субсидию)

Доступ из API Routes через AWS SDK v3 (`@aws-sdk/client-s3`). Файлы никогда не отдаются клиентам напрямую — подписанный URL генерируется на сервере и возвращается на frontend.

## Инфраструктура

**Контейнеризация:** Docker

```
docker-compose.yml
├── app        — Next.js (сборка и сервинг приложения)
├── db         — PostgreSQL
└── (опционально) redis — для очереди задач в более поздних фазах
```

**Конфигурация окружения:** файл `.env` с секретами (не коммитится). Валидируется при запуске через `@t3-oss/env-nextjs` или аналог.

**CI/CD pipeline:**
- Триггер: push в ветку `main` или смёрженный PR
- Шаги:
  1. `tsc --noEmit` — проверка типов TypeScript
  2. `eslint` — линтинг
  3. `prisma migrate deploy` — запуск ожидающих миграций
  4. `next build` — сборка для продакшна
  5. Сборка + push Docker-образа в реестр
  6. Деплой на сервер (через SSH или CLI платформы)
- Инструмент: GitHub Actions (`.github/workflows/deploy.yml`)

**Миграции БД в CI:**
- `prisma migrate deploy` запускается против продакшн-базы до старта нового образа
- Миграции должны быть обратно совместимы (нельзя DROP COLUMN на живых данных)
